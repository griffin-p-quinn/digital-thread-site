System overview

Mendix + NX at 200-user scale

A reference architecture for delivering controlled, parameterized CAD changes to business users at roughly 200 named users while preserving engineering governance, license efficiency, and system reliability. It uses a three-plane model that separates workflow orchestration, deterministic NX execution, and PLM record management.

Architecture principle

Three planes, one control point

The system is divided into three clearly bounded planes. Each plane has a single owner, a clear data contract, and an explicit handoff. The template registry sits at the boundary between the workflow and execution planes — it is a mandatory control point that every job must pass through.

Plane 1 — Workflow
Mendix + stateless AWS services. Owns user interaction, approvals, orchestration, and monitoring.
Plane 2 — Execution
NX on governed EC2 worker slots. Owns deterministic CAD regeneration and artifact production.
Plane 3 — PLM
Teamcenter as system of record. Owns released objects, revisions, and BOM relationships.
Governance boundary
Template registry + admission control. No job enters the execution plane without passing both gates.

Key design decisions

Why these choices matter at 200 users

At 200 concurrent users, NX license tokens become the binding constraint — not queue depth, not CPU. Every design decision flows from this. The license controller sits in the hot dispatch path specifically to prevent token exhaustion from causing silent job failures or indefinite queuing.

Separating the workflow plane from the execution plane allows Mendix to iterate continuously without forcing NX worker restarts or template validation cycles. The slower governed train for the execution plane is intentional: a bad rollout here can invalidate outputs, waste license capacity, and break traceability across released artifacts.

Autoscaling signal
Queue depth + license headroom. Upper bound is license capacity, not queue depth alone.
Retry policy
Recoverable failures retry within envelope. Non-recoverable go to DLQ immediately — no token burning.
Template governance
Signed bundles. Compatibility matrix. Controlled rollback. No ad-hoc template deployment.
Target SLO
95% of standard jobs completed within agreed turnaround. Dead-letter routing for non-recoverable errors.

Job execution sequence

Five deterministic steps per job

1) SQS dequeue, 2) synchronous license gate, 3) slot assignment, 4) NX execution, 5) artifact upload. The worker manager holds SQS visibility until Redis confirms token availability.

Key AWS decisions

NX workers use EC2 Windows Server, not Fargate — NX requires a licensed Windows session with persistent process state. The control plane is Fargate (stateless). Redis on ElastiCache for the license token semaphore: atomic decrements, sub-ms reads, TTL-based slot leases that auto-expire on crash. EventBridge decouples execution from workflow — no direct API dependency.

The edge blend question answered

How a new NX operation gets added

The answer depends on where the new operation lands in the stack. There are three cases, ordered from fastest to most involved.

Case 1 — already exposed
Edge blend method already registered in the facade. Only a new template bundle and schema update needed. No worker change. Scenario B.
Case 2 — dynamic handler path
Facade transport can carry it but the handler isn't written yet. New signed .NET assembly hot-loaded into the dynamic handler host. No NX restart on steady-state workers. Scenario C.
Case 3 — facade DLL change
New IPC message type or shared memory contract needed. Full DLL rebuild, new ECR image, canary pool spinup, NX process restart per slot, regression pack, rolling promotion. Scenario D.
Safe rollout pattern (scenario D)

1. Rebuild and push new worker image to ECR. 2. Update capability registry compatibility matrix. 3. Spin up canary pool with new image — stable pool stays live. 4. Drain in-flight jobs on canary slots, stop NX host, replace DLL, restart NX host, signal ready. 5. Run full NX regression pack on canary. 6. Rolling replace stable pool one slot at a time. 7. Keep old pool warm for 24-hour rollback window. 8. Mark old facade version deprecated in registry.

Release trains & capability model

The system runs two release trains: a fast train for Mendix and the control plane (continuous CI/CD, no NX involvement) and a slow governed train for the NX execution plane (versioned, signed, regression-tested). The capability registry is the coordination point between them — it stores the compatibility matrix and ensures jobs are only routed to worker pools that carry the required capability set.
