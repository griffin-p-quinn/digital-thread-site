Beat 1 / 5
Start with Teamcenter truth, not architecture optimism
The project reset when the ask changed from "is the API up?" to "show me a real graphmart that pulls meaningful Teamcenter data and explain exactly what still does not work." That forced the workflow to anchor on live payloads, tested IDs, and specific missing semantics instead of vague availability checks.

Beat 1 / 5

Start with source truth

The team already had payload bundles and MCP access. The real move was to stop treating those as background context and instead make them the explicit benchmark. That is how the conversation became concrete: which relation families are real, which UIDs resolve, and which business questions are actually answerable?

That baseline made later graph claims falsifiable. If the graph could not show the same revision and dataset trail the Teamcenter payloads showed, the graph did not get credit.

Beat 2 / 5

Flatten without throwing meaning away

An early troubleshooting move changed POST to GET and flattened response shape, which unlocked data flow but risked losing the semantics that make the graph useful. The fix was not "more flattening." The fix was to flatten deliberately: carry through predicate_raw, relation_family, dataset_type, file_count, source_tc_uid, target_tc_uid, and SOA-derived naming fields.

That is the difference between a technically connected graph and a semantically useful graph.

Enrichment fields verified:
- predicate_raw
- relation_family
- dataset_type
- file_count
- source_tc_uid
- target_tc_uid
Beat 3 / 5

Prove that the graph tells a better story

The graph became compelling once it could do more than mirror object existence. Adding revision-to-revision links, change impacts, and workflow targets gave the graph business context. Now it can answer not just "what files are attached" but also "is this revision under change?" and "is there workflow around it?"

That is the point where the graph stopped being an experiment and became a reasoning surface.

Beat 4 / 5

Teach the agent when not to use the graph

A strong demo is not "graph for everything." It is disciplined routing. Relationship-heavy asks go to SPARQL first. Runtime BOM expansion, file bytes, and mutation flows stay in Teamcenter MCP. That clarity is what makes the architecture believable instead of fashionable.

In other words: the win is not replacing Teamcenter. The win is narrowing the expensive Teamcenter calls to the moments that truly require them.

Graph-first asks:
- What revisions exist?
- What datasets are attached?
- Is this revision affected by change?
- Is there workflow around this object?

MCP asks:
- Expand BOM
- Download JT file
- Create / revise / promote
Beat 5 / 5

End with a demo story people can remember

The strongest story is simple: a user asks about an item, the graph finds its revisions and datasets immediately, the agent can surface change and workflow context, and only then - when the ask becomes operational - does Teamcenter MCP enter the loop.

That is why this is demo-ready. It has a narrow, explainable success path grounded in real data and clear architectural boundaries.
