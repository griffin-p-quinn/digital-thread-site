01 · Boundary

Problem

A useful local assistant needs access to files, processes, browsers, builds, and developer tools.

Giving every capability permanent, unrestricted authority would make the assistant difficult to reason about and unsafe to hand to a broader workflow. The design question was specific: how can Maia inspect and test a local Mendix workspace while keeping file, process, browser, and network authority explicit?

02 · System design

My contribution

I built the TypeScript MCP service and its native Windows control center.

I organized the local operations into gated capability families and packaged the server as a standalone executable. The surface covers Mendix, Git, widget, build, task, browser, and documentation work rather than a single demonstration command.

03 · Source record

Proof

  • README.md defines Maia Local MCP, its capability-gate safety model, and the Mendix, Git, widget, task, browser, and documentation tool families.
  • package.json confirms the TypeScript/Node package and its release scripts.
  • Portfolio preview uses a deterministic, sanitized control-center reconstruction; it does not touch a visitor's filesystem, process table, network, or credential store.

04 · Outcome

Result

The project turns a wide set of local developer operations into one inspectable MCP surface.

It preserves an explicit boundary around what the agent may do. It is a packaged local tool, not a claim that arbitrary machine access is safe by default.
