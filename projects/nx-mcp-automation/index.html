01 · Interface gap

Problem

NX automation is powerful, but the host API is not naturally shaped for an LLM agent.

Conversational intent still has to become explicit, typed operations that execute inside the NX runtime and return inspectable results. The project asks what becomes possible when an agent can drive NX directly—extruding, booleaning, and reading FEA output—through Model Context Protocol.

02 · Tool engineering

My contribution

I designed and implemented the MCP-to-NXOpen boundary as 68 typed tools.

Role of generative AI. An AI client can plan and call the typed operations; I supply the CAD intent, parameter boundaries, NX runtime, and engineering review of the generated result.

Geometry creation, feature work, and analysis inspection have distinct, callable contracts. I also built the verification workflow around the hosted NX/remoting architecture rather than presenting a browser mockup as the CAD integration itself.

04 · Outcome

Result

The server makes a substantial NX automation surface addressable through stable tool contracts.

It demonstrates real CAD-host integration and leaves the execution boundary visible: the agent chooses tools, while NXOpen remains the system performing and returning the engineering work.
