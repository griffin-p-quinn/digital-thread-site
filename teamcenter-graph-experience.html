System Overview

From Teamcenter records to agent-ready graph reasoning

We now have a small but meaningful proof of concept graphmart story: Items, revisions, datasets, related revisions, change impacts, and workflow targets all survive into a structure that an agent can traverse in milliseconds. The graph is not replacing Teamcenter - it is becoming the fast semantic layer in front of it.

Why this matters

Graph first for relationships, MCP when the question demands live state

The demo story is no longer "an API is up." It is "a user asks a relationship-heavy PLM question and the agent knows when a graph query is enough, when it should enrich from Teamcenter, and why."

User: "Show me every revision of item 001024 and which datasets are attached." Agent route: 1. Resolve item by graph identity and display attributes 2. Traverse ITEM_HAS_REVISION 3. Traverse REVISION_HAS_DATASET + REVISION_HAS_SPECIFICATION_DATASET 4. Return answer immediately 5. Only call MCP if user then asks to download or mutate something

Decision Engine

What causes the agent to choose graph vs MCP?

The agent is not choosing interfaces randomly. It is choosing based on data shape, freshness requirements, and whether the user asked a relationship question or an operational question.

Graph first
"Show me every revision for item 001024 and which datasets each revision carries."

This is a relationship-traversal question. The graph already stores the structure, so a SPARQL query is the fastest, safest answer.

95%
SELECT ?rev ?dataset ?datasetType WHERE { tc:hasRevision ?rev . OPTIONAL { ?rev ?p ?dataset . VALUES ?p { tc:hasDataset tc:hasSpecificationDataset } ?dataset tc:dataset_type ?datasetType . } }
Graph first
"Are there any changes affecting this revision?"

Now that CHANGE_AFFECTS_REVISION exists, the graph can answer impact questions without a deep change-management crawl.

90%
SELECT ?change ?name WHERE { ?change tc:affectsRevision ; tc:object_name ?name . }
Hybrid
"Who has the latest revision checked out right now?"

The graph may hold last-known checkout metadata, but checkout state is volatile. The agent uses graph for identity and latest revision, then confirms via MCP.

68%
Graph step: SELECT ?rev ?user WHERE { tc:hasRevision ?rev . OPTIONAL { ?rev tc:checked_out_user ?user } } If stale or missing -> MCP getProperties(revisionUid)
MCP direct
"Download the JT file and expand the BOM under this assembly."

These are runtime or binary requests. The graph helps locate the right object, but the actual action happens through Teamcenter native services.

20%
Graph locate: SELECT ?dataset WHERE { tc:hasDataset ?dataset . ?dataset tc:dataset_type "IMAN_Rendering" . } Then MCP: - getDatasetFiles(datasetUid) - expandPSOneLevel(itemRevisionUid)
Routing rule of thumb
Graph

Ask contains relationships, lineage, impact, attachment context, or filterable business structure. Use SPARQL first.

Hybrid

Ask starts with structure but ends with volatile operational state. Use graph for fast narrowing, MCP for confirmation.

MCP

Ask needs file bytes, runtime BOM, write operations, or live workflow task detail. Route directly to Teamcenter.

SPARQL traversal proof

Item → Revision → Dataset traversal query

Full multi-graph SPARQL queries traverse the pipeline data end-to-end. This query joins edges and node properties across named graphs to resolve the complete Item → Revision → Dataset chain.

PREFIX tc: <http://proxi.demo/tc/ontology#> SELECT ?itemName ?revName ?dsName ?dsType WHERE { GRAPH <http://proxi.demo/tc/edges> { ?item tc:hasRevision ?rev . ?rev tc:hasDataset ?ds . } GRAPH <http://proxi.demo/tc/nodes> { ?item a tc:Item ; tc:objectName ?itemName . ?rev a tc:ItemRevision ; tc:objectName ?revName . ?ds a tc:Dataset ; tc:datasetName ?dsName ; tc:datasetType ?dsType . } } LIMIT 15
